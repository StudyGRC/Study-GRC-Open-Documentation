### 1. Purpose
The purpose of this procedure is to mandate and standardize the implementation of Multi-Factor Authentication (MFA) across all Study GRC Inc. systems. This procedure operationalizes the *Access Control Policy* and aligns with **CISA Cybersecurity Performance Goal (CPG) 2.1** and **NIST SP 800-63B**, ensuring that compromised passwords do not lead to unauthorized access, data breaches, or financial loss.

### 2. Scope
This procedure applies to:
*   **Personnel:** All Board Directors, Officers, employees, contractors, and volunteers.
*   **Systems:** All Study GRC Inc. information systems, including but not limited to Google Workspace (Identity Provider), Slack, GitHub, AWS, financial platforms (QuickBooks, Bank Accounts), and social media accounts.
*   **Devices:** All personal and organization-owned devices used to access organizational data.

**Exclusions:** Systems that do not support MFA must be documented in the *Risk Register* and require a specific *MFA Exemption Waiver* approved by the CISO.

### 3. Definitions

| Term | Definition |
| :--- | :--- |
| **MFA (Multi-Factor Authentication)** | A security mechanism requiring two or more separate types of credentials (factors) for authentication (e.g., something you know + something you have). |
| **TOTP** | Time-based One-Time Password. A temporary code generated by an app (e.g., Google Authenticator, 1Password) that changes every 30 seconds. |
| **FIDO2 / WebAuthn** | An open authentication standard that enables passwordless authentication using hardware keys (e.g., YubiKey) or platform authenticators (e.g., TouchID, Windows Hello). |
| **Recovery Codes** | A set of one-time-use codes generated during MFA setup to regain access if the primary MFA device is lost. |
| **IdP (Identity Provider)** | The centralized system used to manage user identities (Study GRC utilizes Google Workspace as the primary IdP). |

### 4. Procedures

#### 4.1 Approved MFA Factor Hierarchy
Users must select the strongest available factor supported by the system.

*   **Tier 1 (Preferred/Mandatory for Admins & Finance):** FIDO2 Hardware Keys (e.g., YubiKey) or Passkeys.
*   **Tier 2 (Standard):** Mobile App TOTP (e.g., Google Authenticator, Authy, 1Password, Bitwarden).
*   **Tier 3 (Restricted):** SMS/Text Message or Voice Call.
    *   *Restriction:* SMS MFA is **prohibited** for Board Members, C-Level Officers, and IT Administrators unless no other option exists.

#### 4.2 General Enrollment Workflow
All users must complete the following steps within **48 hours** of account provisioning:

1.  **Login:** Access the security settings of the target application.
2.  **Select Factor:** Choose "Two-Step Verification" or "MFA."
3.  **Configuration:**
    *   Scan the QR code using an approved TOTP app or insert the Hardware Key.
    *   Verify the connection by entering the generated code or tapping the key.
4.  **Backup Configuration (Mandatory):**
    *   Generate **Backup/Recovery Codes**.
    *   Save these codes immediately into the organization-provisioned Password Manager (e.g., 1Password/Bitwarden) in the "Secure Notes" section.
    *   *Prohibited:* Do not save recovery codes in plain text files, screenshots on the desktop, or physical sticky notes.

#### 4.3 Google Workspace Configuration (Primary IdP)
As Study GRC’s primary Identity Provider, Google Workspace MFA protects access to email, drive, and SSO-connected apps.

1.  Navigate to `myaccount.google.com` -> **Security**.
2.  Select **2-Step Verification**.
3.  **Step 1:** Configure the primary method (Google Prompt or Authenticator App).
4.  **Step 2:** Scroll to "Backup codes" -> **Get backup codes**.
5.  **Step 3:** Copy the 10 codes and store them in the Password Manager.
6.  **Step 4:** (Optional but Recommended) Add a secondary phone number for recovery, provided it is a personal device not shared with others.

#### 4.4 High-Privilege Account Configuration
Users with **Super Admin**, **Billing Admin**, or **Bank Access** roles must:

1.  Register at least **two** distinct hardware keys (Primary and Backup) where supported.
2.  Disable SMS/Voice fallbacks if the platform allows, to prevent SIM-swapping attacks.
3.  Undergo a witnessed setup session (screen share) with the CISO or IT Delegate to verify configuration.

#### 4.5 MFA for Shared Accounts
Shared accounts (e.g., `info@studygrc.org` or social media handles) are generally discouraged but, if necessary, must be secured as follows:

1.  **Do not** tie MFA to a specific individual’s personal phone number.
2.  **Procedure:**
    *   Store the TOTP "Secret Key" (the text string usually accompanying the QR code) in the organization’s shared Password Manager vault.
    *   Authorized users access the OTP code directly from the Password Manager interface.

#### 4.6 Lost Device and Reset Protocol
If a user loses their MFA device:

1.  **Immediate Action:** User must use their stored Recovery Codes to access the account and revoke the lost device's trust immediately.
2.  **Support Request:** If Recovery Codes are unavailable, the user must contact IT Support.
3.  **Identity Verification:** Before resetting MFA, IT Support must verify identity via:
    *   Video call verification (comparing against ID on file).
    *   Verification by a Manager or Director.
4.  **Reset:** IT Admin generates a temporary bypass code valid for 1 hour to allow reconfiguration.

### 5. Roles and Responsibilities

| Role | Responsibility |
| :--- | :--- |
| **All Users** | Enroll in MFA upon account creation; store recovery codes securely; report lost devices immediately. |
| **IT/Security Team** | Enforce MFA policies via technical controls (e.g., "Enforce" mode in Google Admin); assist with resets; audit compliance. |
| **CISO** | Review and approve MFA Exemption Waivers; update approved factor lists based on emerging threats. |
| **Hiring Manager** | Ensure new hires are informed of the MFA requirement during onboarding. |

### 6. Compliance and Enforcement
**Technical Enforcement:** Systems capable of enforcing MFA (e.g., Google Workspace, Slack, GitHub Organization) will be configured to **"Hard Fail"**—users without MFA configured will be locked out after the grace period.

**Disciplinary Action:** Failure to comply with this procedure, including disabling MFA or sharing OTP codes with others, is a violation of the *Master Information Security Policy*. Consequences may include temporary account suspension, revocation of administrative privileges, or termination of employment/volunteer status.
